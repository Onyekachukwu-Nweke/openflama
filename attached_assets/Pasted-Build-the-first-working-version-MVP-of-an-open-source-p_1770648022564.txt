Build the first working version (MVP) of an open-source project called OpenFlama: an agentic finance runtime + forecasting engine designed to power future “financial agents” and integrate later into Agentfunds.io (my other Replit webapp, we will examinate this later).
Product Definition (Scope)
Flama is NOT a new foundation model trained from scratch. It is a platform that:
1.	Reuses an open backbone LLM (e.g., Llama/Mistral/Qwen via Ollama or llama.cpp) for planning + tool use + explanations.
2.	Provides a quant forecasting engine (time-series prediction) that produces:
o	point forecast (expected return / price change)
o	prediction interval (uncertainty)
o	explanation and trace (what data/features/models were used)
3.	Provides benchmarks (reproducible walk-forward evaluation + baselines).
4.	Includes docs + a minimal website to encourage GitHub contributions.
Tech Constraints
•	Use TypeScript + Python + SQL.
•	Keep it cost-effective: local-first dev, CPU-friendly defaults.
•	Provide clean modular architecture so LLM can be swapped easily.
Repo Structure (Monorepo)
Create a monorepo with:
•	/apps/api (Python FastAPI): prediction, backtest, datasets, model registry.
•	/apps/web (TypeScript React or Next.js): minimal landing + dashboard.
•	/packages/sdk-ts (TypeScript client SDK): typed API client for Agentfunds.io integration.
•	/packages/core-py (Python library): forecasting engine + backtesting harness.
•	/docs (Markdown): quickstart, architecture, contributing, benchmark methodology.
•	/infra (docker-compose): Postgres + (optional) Redis.
Include:
•	.env.example
•	README.md with quickstart + screenshots placeholders
•	LICENSE (MIT or Apache-2.0)
•	basic CI (lint/test) if feasible in this environment
MVP Features (Must Have)
1) FastAPI Backend
Implement endpoints:
•	POST /v1/predict
o	input: ticker, horizon (e.g., 1d/5d/20d), as_of_date, target (return or price), optional features flags
o	output: forecast, interval_low/high, model_used, features_used, data_window, trace_id
•	POST /v1/backtest
o	input: ticker(s), horizon, start/end, baseline selection
o	output: metrics (MAE/MAPE, directional accuracy, hit-rate), plots links or JSON series
•	GET /v1/models (list available forecasting models/baselines)
•	GET /v1/health
Data layer:
•	Use Postgres with a simple schema for:
o	predictions (inputs, outputs, timestamps)
o	backtest runs (config, metrics, status)
o	datasets metadata
If Postgres is too heavy for first run, allow SQLite fallback but keep Postgres-ready schema.
2) Forecasting Engine (Baselines First)
In /packages/core-py:
•	Implement baselines:
o	Random Walk (no-change)
o	Simple Momentum (e.g., mean of last N returns)
o	ARIMA or ETS (if available; if not, keep as optional)
•	Add a strong tabular baseline:
o	LightGBM or XGBoost (whichever is easiest) on engineered features:
	lagged returns, rolling volatility, rolling mean, RSI-like feature
•	Implement walk-forward evaluation to avoid leakage.
•	Output prediction intervals (simple approach ok: residual quantiles, bootstrap, or conformal-lite).
3) Agentic Runtime (LLM as Orchestrator)
Do NOT train a model now. Instead:
•	Create a pluggable interface LLMProvider:
o	OllamaProvider (local)
o	OpenAIProvider placeholder (do not require keys)
•	Implement a minimal “planner” flow:
o	User asks: “Predict AAPL 5d”
o	LLM decides: call /features (internal), then /predict, then returns a human-readable explanation + risks.
If full tool-calling is too much for MVP, implement it as a structured prompt template and deterministic routing, but keep interfaces ready for real tool-calling later.
Security note: never expose Ollama host publicly by default; bind to localhost and document safe config.
4) Web App (Minimal but Real)
In /apps/web:
•	Landing page with:
o	What is Flama (Financial Large-Language Model Agora)
o	Design using Black predominant layout + red elements. Similar to: openclaw.ai
o	Quickstart
o	Benchmark philosophy
o	GitHub/Contributing links
•	Simple dashboard page:
o	form: ticker + horizon
o	shows: forecast + interval + explanation
o	shows: recent predictions table from API
5) TypeScript SDK
In /packages/sdk-ts:
•	Provide createFlamaClient({ baseUrl, apiKey? })
•	Methods: predict(), backtest(), listModels()
Nice-to-Have (If Time)
•	Background jobs for backtests (Celery + Redis)
•	Simple caching
•	Export results to CSV/JSON
•	A “leaderboard” JSON produced by backtest harness
Quality Requirements / Acceptance Criteria
•	Running locally should be 3 commands max:
a.	start db (optional)
b.	start api
c.	start web
•	POST /v1/predict works end-to-end using at least one baseline model.
•	POST /v1/backtest produces metrics for at least one ticker + one baseline.
•	Code is modular: forecasting engine independent from LLM provider.
•	Docs explain:
o	how to run
o	how evaluation avoids lookahead bias
o	how to contribute a new model or feature
